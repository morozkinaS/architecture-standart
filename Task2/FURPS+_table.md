# FURPS+ Таблица требований к MVP по открытию депозитов онлайн

| Категория | Требование | Обоснование / Комментарий |
|----------|------------|---------------------------|
| **F** | Клиент может подать заявку на депозит через сайт, указав ФИО и номер телефона | Это базовый MVP-сценарий для привлечения новых клиентов |
| **F** | Клиент может подать заявку на депозит через интернет-банк, указав счёт и сумму | Центральный функционал для цифровизации |
| **F** | Система отображает актуальные ставки по депозитам на сайте и в интернет-банке | Без актуальных ставок — нет продаж |
| **F** | В интернет-банке ставки персонализированы для клиента (на основе его истории) | Увеличивает конверсию и доверие |
| **F** | После подачи заявки в интернет-банке клиент получает СМС-код для подтверждения | Требование безопасности и подтверждения действия |
| **F** | Система автоматически рассчитывает ставку по депозиту на основе данных о кредитном риске и рыночных условиях | Замена ручного Excel-расчёта — ключевое улучшение |
| **F** | Ставки рассчитываются и хранятся в централизованном источнике (АБС или новом сервисе), доступном для бэк-офиса депозитов и кредитов | Устраняет разрозненность данных и ошибки в Excel |
| **F** | При подаче заявки через сайт, система передаёт данные в систему кол-центра для последующего звонка | Обеспечивает непрерывность процесса |
| **F** | Менеджер бэк-офиса может просмотреть заявку в АБС, утвердить ставку и инициировать открытие депозита | Сохраняется текущий workflow для MVP |
| **U** | Интерфейс сайта и интернет-банка соответствует корпоративному стилю (цвета, шрифты, логотипы) | Повышает доверие и узнаваемость бренда |
| **U** | Форма подачи заявки проста: максимум 3 поля (ФИО, телефон, сумма) | Для пожилых клиентов — минимальная когнитивная нагрузка |
| **U** | Отклик интерфейса — менее 500 мс на все действия (загрузка ставок, отправка заявки) | Сейчас загрузка справочников >1 сек — критично |
| **R** | Система должна быть доступна 99.9% времени в течение года | Требование к банковским сервисам |
| **R** | В случае сбоя в основном ЦОД — автоматическое переключение на резервный ЦОД | Уже есть инфраструктура — нужно её задействовать |
| **R** | Данные о заявках не теряются при сбоях (гарантированная доставка) | Клиент не должен терять заявку из-за технической ошибки |
| **P** | Время отклика при загрузке списка депозитов — < 500 мс | Сейчас >1 сек — недопустимо для UX |
| **P** | Система должна выдерживать нагрузку до 500 заявок/день (в MVP) и масштабироваться до 2000/день | Прогнозируемый рост после маркетинговой кампании |
| **P** | СМС-отправка — < 10 сек после подтверждения | Клиент ждёт подтверждения — задержка снижает доверие |
| **S** | Все изменения в системах документируются (архитектурные решения, API, конфигурации) | Для будущей поддержки и аудита |
| **S** | Используются технологии, в которых есть экспертиза в банке (.NET, SQL, Oracle, React) | Снижает риски и время на обучение |
| **S** | Архитектура позволяет в будущем перевести интернет-банк на микросервисы без полного переписывания | Позволит масштабировать и заменить АБС-интеграцию |
| **S** | Возможность подключения новых каналов (например, мобильное приложение) без перестройки ядра | Гибкость для будущих итераций |
| **S** | Интеграция с АБС осуществляется через API (не через прямой доступ к БД) | Устраняет риск повреждения данных и повышает стабильность |
| **S** | Использование Kafka в будущем — планируется как архитектурный шаг, но не в MVP | Для масштабирования обмена сообщениями между сервисами |
| **S** | Резервное копирование данных заявок — ежедневное, с хранением 90 дней | Требования регуляторов и внутренней политики |
| **+** | Все данные передаются по HTTPS (TLS 1.2+) | Защита персональных данных — требование ФЗ-152 и PCI DSS |
| **+** | Данные клиентов шифруются при хранении в базах (PII-данные) | Дополнительная мера безопасности |
| **+** | Доступ к данным ставок и заявкам — только для авторизованных пользователей (роли: менеджеры депозитов, кредитов, операторы) | Разграничение доступа — требование безопасности |
| **+** | Сайт и интернет-банк совместимы с современными браузерами (Chrome, Edge, Safari, Firefox) | Для охвата широкой аудитории, включая пожилых |
| **+** | Система не требует установки плагинов или дополнительного ПО на стороне клиента | Ключевое для пожилых пользователей |
| **+** | Интеграция с АБС не увеличивает нагрузку на её базу данных | АБС уже перегружена — нельзя писать напрямую в Oracle |
| **+** | Новая функциональность реализуется без изменения ядра интернет-банка (если возможно) | Чтобы не зависеть от подрядчика по обновлению ядра |
| **+** | Использование MS SQL для хранения заявок (а не Oracle) — предпочтительно | Команда IT имеет опыт с MS SQL — проще и дешевле |
| **+** | Система должна позволять в будущем расширить функционал на кредиты без переделки архитектуры | Прогнозируемое развитие: кредиты — следующий этап |
